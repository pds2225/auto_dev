# Prompt Templates

Use these templates for the selected single task only. Choose the target AI explicitly when the user is coordinating Kimi, Claude, Cursor, and Codex.

## Target AI Selector

- Kimi: diagnosis and idea generation only; no code edits.
- Claude: review Kimi plans, reduce scope, reason about complex logic, define tests.
- Cursor: find code locations and make narrow UI or bug patches.
- Codex: apply local repo changes, run commands, verify, commit, PR, Notion, and skill maintenance.

## 오류 해결용 구현 프롬프트

```text
대상 AI:
[Cursor / Claude / Codex 중 하나]

역할:
너는 기존 코드베이스에서 오류 1개만 최소 변경으로 고치는 구현자다.

작업 목표:
[무엇을 고치는가]

배경:
[왜 고치는가. 사용자에게 보이는 증상과 발생 가능한 문제를 2-3줄로 설명]

수정 가능 범위:
- [어떤 파일 또는 폴더]
- [관련 테스트 파일]

수정 금지 범위:
- 전체 리팩토링 금지
- 요구사항 변경 금지
- DB/API 계약 변경 금지
- 관련 없는 파일 수정 금지
- Secret 값 출력 또는 하드코딩 금지

구현 요구사항:
1. 오류 발생 원인을 먼저 확인한다.
2. 확인된 원인에 필요한 최소 코드만 수정한다.
3. 예외/빈 상태/실패 상태를 사용자가 이해할 수 있게 처리한다.
4. 기존 정상 흐름은 유지한다.

검증 방법:
1. [실행 명령어]
2. [테스트 명령어]
3. [회귀 확인 포인트]

출력 형식:
1. 원인
2. 수정 파일
3. 수정 내용
4. 테스트 결과
5. 회귀 위험
6. 다음 작업 1개
```

## 기능 개선용 구현 프롬프트

```text
대상 AI:
[Cursor / Claude / Codex 중 하나]

역할:
너는 기존 구조를 유지하면서 사용자 가치가 큰 기능 개선 1개만 구현하는 개발자다.

작업 목표:
[무엇을 개선하는가]

배경:
[왜 필요한가. 사용자가 얻는 이점을 쉬운 말로 설명]

수정 가능 범위:
- [어떤 파일 또는 폴더]
- [필요한 테스트 파일]

수정 금지 범위:
- 새 대형 기능 추가 금지
- 제품 방향 변경 금지
- DB/API 계약 변경 금지
- 전체 리팩토링 금지
- 관련 없는 UI/문구/설정 변경 금지

구현 요구사항:
1. 현재 동작을 먼저 확인한다.
2. 개선 범위를 한 가지 사용자 흐름으로 제한한다.
3. 입력/성공/실패/빈 상태를 처리한다.
4. 기존 테스트가 있으면 회귀 테스트를 추가하거나 갱신한다.

검증 방법:
1. [실행 명령어]
2. [테스트 명령어]
3. [사용자 흐름 확인 방법]

출력 형식:
1. 변경 요약
2. 수정 파일
3. 테스트 결과
4. 회귀 체크포인트
5. 다음 작업 1개
```

## 프론트엔드 개선용 구현 프롬프트

```text
대상 AI:
[Cursor / Codex 중 하나. 화면 구조 판단이 복잡하면 Claude 검토 먼저]

역할:
너는 기존 화면을 유지하면서 프론트엔드 문제 1개만 개선하는 구현자다.

작업 목표:
[어떤 화면/흐름을 개선하는가]

배경:
[사용자가 어디서 헷갈리는지, 어떤 문제가 생기는지 설명]

수정 가능 범위:
- [관련 컴포넌트 파일]
- [관련 CSS/스타일 파일]
- [관련 테스트 파일]

수정 금지 범위:
- 디자인 전면 개편 금지
- API 수정 금지
- 라우팅 구조 변경 금지
- 관련 없는 화면 수정 금지
- 카드/장식 요소 과다 추가 금지

구현 요구사항:
1. 데스크톱과 모바일에서 모두 깨지지 않게 한다.
2. 버튼, 입력폼, 결과, 에러, 빈 상태 중 이번 문제와 관련된 상태를 처리한다.
3. 긴 텍스트가 버튼/카드/패널 밖으로 넘치지 않게 한다.
4. 기존 디자인 시스템과 컴포넌트 패턴을 따른다.

검증 방법:
1. [개발 서버 실행 명령어]
2. [화면 확인 URL 또는 경로]
3. [모바일/데스크톱 확인 항목]
4. [테스트 명령어]

출력 형식:
1. 변경 요약
2. 수정 파일
3. 화면 확인 방법
4. 테스트 결과
5. 회귀 위험
6. 다음 작업 1개
```
